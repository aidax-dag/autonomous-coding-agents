## 1. 요구사항 (Requirements)

### 1.1 배경

- E04는 런타임 신뢰성, 정책 강제, 비용 통제를 동시에 보장해 Builder v1의 운영 가능성을 확보하는 핵심 계층이다.

### 1.2 문제

- 중복 실행/재시도 통제가 없거나 정책 평가가 누락되면 보안 위반과 비용 폭증이 발생한다.

### 1.3 목표

- 비용 사용량과 절감 효과를 조회하는 API를 추가한다.

### 1.4 범위 / 비범위

- 범위: 기간/티켓/에이전트별 집계, 예산 대비 사용률, 절감 추정
- 비범위: 대시보드 고도화

### 1.5 제약사항

- 게이트 순서 준수: 요구사항 -> 기능정의 -> 설계 -> 스펙정의 -> 구현 -> 테스트 -> 결과정리 -> 요구사항 업데이트
- 설계/스펙 승인 전 구현 금지

## 2. 기능정의 (Feature Definition)

### 2.1 구현 항목

1. TASK2에서 정의된 T0409 항목을 v1 범위 기준으로 구현한다.
2. 산출물을 문서/코드/테스트로 남긴다.
3. 후속 티켓이 재사용 가능한 형태로 결과를 남긴다.

### 2.2 완료 조건

1. 산출물 링크가 본문에 업데이트되어야 한다.
2. 상태가 reviewing으로 전이되어야 한다.
3. 리뷰에서 approved 1회 이상을 받아야 한다.

## 3. 설계 (Design)

### 3.1 설계 요약

- 집계 쿼리와 상세 내역 조회를 분리한 API 설계

### 3.2 보안/정책 포인트

- 정책 판단 근거와 비용 계산 근거를 감사 가능한 로그로 남긴다.
- fail-open 대신 fail-safe 원칙으로 정책 위반 가능 상태를 기본 차단한다.

### 4.3 합격 조건 (Acceptance Criteria)

1. 정의된 인터페이스/스키마가 문서와 코드에 일치한다.
